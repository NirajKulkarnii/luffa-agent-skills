# Luffa Agent Payments Skill

Provides non-custodial infrastructure for AI agents to autonomously execute
crypto payments on behalf of users — within strictly user-defined limits and
protected by TOTP (Time-Based One-Time Password) authentication.

## How TOTP Protection Works

Every payment requires a 6-digit TOTP code generated by the user's
authenticator app (Google Authenticator, Authy, etc.). The agent:

- ✅ CAN prepare a transaction
- ✅ CAN request a TOTP challenge from the server
- ✅ CAN submit the payment once the user provides the code
- ❌ CANNOT generate a TOTP code
- ❌ CANNOT approve its own payment
- ❌ CANNOT intercept or replay a code (codes expire in 30 seconds)

Even if the agent is fully compromised, it cannot complete a payment
without a valid TOTP code from the user's physical device.

**Base URL**: `https://api.luffa.im/v1`

**Auth**: Bearer token via `Authorization` header.

## Authentication & Credentials

Read credentials from environment variables:
- `LUFFA_API_KEY` → API key from the developer portal
- `LUFFA_SECRET_KEY` → Secret key for signing requests

**Never** output credentials to logs, response content, or any
user-visible interface.
```typescript
const BASE_URL = 'https://api.luffa.im/v1';
const LUFFA_API_KEY = process.env.LUFFA_API_KEY || 'demo-key';

async function luffaFetch(method: 'GET' | 'POST' | 'DELETE', path: string, body?: object) {
  const timestamp = Date.now().toString();
  const headers: Record<string, string> = {
    'X-Luffa-API-Key': LUFFA_API_KEY,
    'X-Luffa-Timestamp': timestamp,
    'Content-Type': 'application/json',
  };
  const res = await fetch(`${BASE_URL}${path}`, {
    method,
    headers,
    ...(body && { body: JSON.stringify(body) }),
  });
  if (!res.ok) throw new Error(`Luffa API error: ${res.status} ${res.statusText}`);
  return res.json();
}
```

## API Reference

### 1. POST /agent/totp/setup

One-time enrollment. Generates a TOTP secret and QR code for the user to
scan into their authenticator app. Must be completed before any agent
payments can be authorized.

**Request Body**:

| Param | Type | Required | Description |
|---|---|---|---|
| `userAddress` | string | Yes | Wallet address or Luffa DID |
| `label` | string | No | Label shown in authenticator app (default: "Luffa Agent Payments") |

**Response**:
```json
{
  "code": "0",
  "data": {
    "totpSecret": "JBSWY3DPEHPK3PXP",
    "qrCodeUrl": "https://api.luffa.im/v1/agent/totp/qr?secret=...",
    "backupCodes": [
      "XXXX-XXXX-XXXX",
      "YYYY-YYYY-YYYY",
      "ZZZZ-ZZZZ-ZZZZ"
    ],
    "setupExpiresAt": "2026-03-04T10:30:00Z"
  },
  "msg": "success"
}
```

> Display the QR code to the user immediately. The user must scan it in
> their authenticator app and verify with one valid code before setup
> is confirmed. Store backup codes securely — they are shown only once.

### 3. POST /agent/authorize

Create an agent payment session with defined spending rules and scope.
This does NOT trigger a TOTP check — it only registers the session policy.
TOTP is required at the individual payment level via `/agent/totp/challenge`
and `/agent/pay`.

**Request Body**:

| Param | Type | Required | Description |
|---|---|---|---|
| `agentId` | string | Yes | Unique identifier for the agent |
| `userAddress` | string | Yes | Wallet address of the authorizing user |
| `spendingLimitAmount` | string | Yes | Max total the agent may spend in this session |
| `spendingLimitSymbol` | string | Yes | Token for the limit (e.g. `EDS`, `USDT`) |
| `perTransactionLimit` | string | Yes | Max amount per single payment |
| `network` | string | Yes | Network for all payments in this session |
| `expiresAt` | string | Yes | ISO 8601 session expiry |
| `scope` | string[] | Yes | Allowed payment types e.g. `["service_fee", "swap"]` |
| `requireTotpPerTransaction` | boolean | Yes | If `true`, every payment needs its own TOTP code |
| `description` | string | Yes | Human-readable purpose shown to user |

**Response**:
```json
{
  "code": "0",
  "data": {
    "authorizationId": "auth_abc123",
    "status": "active",
    "spendingLimitAmount": "100.00",
    "perTransactionLimit": "20.00",
    "spendingLimitSymbol": "EDS",
    "requireTotpPerTransaction": true,
    "scope": ["service_fee", "swap"],
    "expiresAt": "2026-03-05T00:00:00Z"
  },
  "msg": "success"
}
```

### 5. POST /agent/totp/challenge

**This is the TOTP gate.** Before every payment, the agent requests a
challenge token from the server. The server sends a push notification to
the user's device displaying the transaction details. The user opens their
authenticator app, reads the 6-digit code, and provides it to the agent.
The agent then submits both the challenge token and the TOTP code together
in `POST /agent/pay`.

The agent never generates, stores, or validates the code itself.

**Request Body**:

| Param | Type | Required | Description |
|---|---|---|---|
| `authorizationId` | string | Yes | Active session ID |
| `agentId` | string | Yes | Agent identifier |
| `to` | string | Yes | Intended recipient address or DID |
| `amount` | string | Yes | Exact payment amount |
| `symbol` | string | Yes | Token symbol |
| `paymentType` | string | Yes | Must match an approved scope value |
| `memo` | string | No | Description of this payment |

**Response**:
```json
{
  "code": "0",
  "data": {
    "challengeToken": "chal_9f8e7d6c5b4a",
    "expiresAt": "2026-03-04T10:05:30Z",
    "expiresInSeconds": 30,
    "transactionSummary": {
      "to": "0xRecipient",
      "amount": "10.00",
      "symbol": "EDS",
      "paymentType": "service_fee",
      "memo": "DEX routing fee"
    },
    "pushNotificationSent": true
  },
  "msg": "success"
}
```

> After receiving the challenge token, present this prompt to the user:
```
┌──────────────────────────────────────────────┐
│  🔐 Agent Payment — TOTP Required            │
│                                              │
│  The agent wants to make this payment:       │
│                                              │
│  To:       @bob / 0xRecipient               │
│  Amount:   10.00 EDS                         │
│  For:      DEX routing fee                   │
│                                              │
│  Open your authenticator app and enter       │
│  the 6-digit code for "Luffa Agent           │
│  Payments":                                  │
│                                              │
│  Code: [ _ ][ _ ][ _ ] [ _ ][ _ ][ _ ]      │
│                                              │
│  ⏱ Code expires in 30 seconds               │
└──────────────────────────────────────────────┘
```

### 6. POST /agent/pay

Submit the payment with both the challenge token and the user's TOTP code.
The server independently verifies the TOTP code against the user's secret —
the agent only passes it through and never validates it locally.

**Request Body**:

| Param | Type | Required | Description |
|---|---|---|---|
| `authorizationId` | string | Yes | Active session ID |
| `agentId` | string | Yes | Agent identifier |
| `challengeToken` | string | Yes | Token from `/agent/totp/challenge` |
| `totpCode` | string | Yes | 6-digit code from user's authenticator app |
| `to` | string | Yes | Recipient address or Luffa DID |
| `amount` | string | Yes | Payment amount — must match challenge exactly |
| `symbol` | string | Yes | Token symbol — must match challenge exactly |
| `network` | string | Yes | Network — must match authorization |
| `paymentType` | string | Yes | Must match an approved scope value |
| `memo` | string | No | Payment description |

**Response**:
```json
{
  "code": "0",
  "data": {
    "txHash": "0x...",
    "status": "confirmed",
    "amount": "10.00",
    "symbol": "EDS",
    "amountSpent": "40.00",
    "amountRemaining": "60.00",
    "authorizationId": "auth_abc123",
    "totpVerified": true
  },
  "msg": "success"
}
```

**Payment rejection reasons**:

| Error | Meaning |
|---|---|
| `TOTP_INVALID` | Code was wrong — ask user to retry |
| `TOTP_EXPIRED` | 30-second window passed — request new challenge |
| `TOTP_ALREADY_USED` | Replay attack detected — request new challenge |
| `CHALLENGE_MISMATCH` | Amount or recipient was altered — reject and alert |
| `LIMIT_EXCEEDED` | Payment exceeds per-transaction or session limit |
| `SCOPE_VIOLATION` | Payment type not in approved scope |
| `SESSION_EXPIRED` | Authorization session has expired |

### 8. POST /agent/revoke

Immediately terminate agent payment authorization.

**Request Body**:

| Param | Type | Required | Description |
|---|---|---|---|
| `authorizationId` | string | Yes | Session to revoke |
| `revokedBy` | string | Yes | `user` or `agent` |
| `totpCode` | string | Conditional | Required if `revokedBy` is `user` |
| `reason` | string | No | Optional revocation reason |

> User-initiated revocations require a TOTP code to prevent a compromised
> agent from revoking its own session to cover tracks.

---

### 9. POST /agent/extend

Request extension of spending limit or expiry. Always triggers a fresh
TOTP challenge — the existing session's codes cannot be reused.

**Request Body**:

| Param | Type | Required | Description |
|---|---|---|---|
| `authorizationId` | string | Yes | Session to extend |
| `newSpendingLimit` | string | No | New total limit (must exceed current) |
| `newExpiresAt` | string | No | New expiry datetime |
| `totpCode` | string | Yes | Fresh code from authenticator app |

## TOTP Security Model
```
┌─────────────────────────────────────────────────────────┐
│                    WHAT THE AGENT SEES                  │
│                                                         │
│  1. Prepares transaction details                        │
│  2. Requests challenge token from server                │
│  3. Displays payment details + code prompt to user      │
│  4. Receives 6-digit code from user                     │
│  5. Submits: challengeToken + totpCode + tx details     │
│  6. Server responds: approved or rejected               │
│                                                         │
│  ❌ Agent never generates a code                        │
│  ❌ Agent never validates a code                        │
│  ❌ Agent never stores the TOTP secret                  │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                   WHAT THE SERVER DOES                  │
│                                                         │
│  1. Receives challengeToken + totpCode                  │
│  2. Looks up user's TOTP secret (stored server-side)    │
│  3. Independently computes expected code for this       │
│     30-second window                                    │
│  4. Compares — rejects if mismatch                      │
│  5. Checks code has not been used before (anti-replay)  │
│  6. Verifies tx details match the original challenge    │
│  7. Only then executes the transaction                  │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                WHAT LIVES ON USER'S DEVICE              │
│                                                         │
│  ✅ TOTP secret (in authenticator app only)             │
│  ✅ Generated 6-digit codes (visible 30s then gone)     │
│  ✅ Backup codes (stored by user only)                  │
│                                                         │
│  Even if agent + server are both compromised,           │
│  no payment can be made without the user's              │
│  physical device.                                       │
└─────────────────────────────────────────────────────────┘
```

## Cross-Skill Workflows

### Workflow A: First-Time Setup → First Payment
```
1. luffa-agent-payments  GET  /agent/totp/status          → not enrolled
2. luffa-agent-payments  POST /agent/totp/setup           → QR code shown to user
3. —                     —                                → user scans QR in authenticator app
4. luffa-agent-payments  POST /agent/authorize            → create session with rules
5. luffa-agent-payments  POST /agent/totp/challenge       → server sends push, shows prompt
6. —                     —                                → user enters 6-digit TOTP code
7. luffa-agent-payments  POST /agent/pay                  → submit with challengeToken + totpCode
8. luffa-messenger        POST /messenger/send            → notify user payment confirmed
```

### Workflow B: Returning Agent — Standard Payment
```
1. luffa-agent-payments  GET  /agent/totp/status          → enrolled ✅
2. luffa-agent-payments  GET  /agent/authorization-status → session active, limit available
3. luffa-agent-payments  POST /agent/totp/challenge       → request challenge for this tx
4. —                     —                                → user reads code from authenticator
5. luffa-agent-payments  POST /agent/pay                  → submit payment
```

### Workflow C: TOTP Code Expired Mid-Flow
```
1. luffa-agent-payments  POST /agent/totp/challenge       → challengeToken issued
2. —                     —                                → user takes >30 seconds
3. luffa-agent-payments  POST /agent/pay                  → fails: TOTP_EXPIRED
4. luffa-agent-payments  POST /agent/totp/challenge       → request fresh challenge
5. —                     —                                → user enters new code promptly
6. luffa-agent-payments  POST /agent/pay                  → success
```

### Workflow D: Compromised Agent Attempt Detected
```
1. Compromised agent     POST /agent/totp/challenge       → gets challengeToken
2. Compromised agent     POST /agent/pay (no totpCode)    → fails: TOTP_INVALID
3. Compromised agent     POST /agent/pay (fake totpCode)  → fails: TOTP_INVALID
4. Server               —                                 → flags repeated failures
5. Server               POST /agent/revoke (auto)         → session terminated
6. luffa-messenger       POST /messenger/send             → alerts user of suspicious activity
```

### Workflow E: Agent Pays for DEX Swap Mid-Task
```
1. luffa-agent-payments  GET  /agent/authorization-status → scope includes "swap" ✅
2. luffa-dex             POST /dex/swap                   → prepare swap
3. luffa-agent-payments  POST /agent/totp/challenge       → show swap fee details to user
4. —                     —                                → user enters TOTP code
5. luffa-agent-payments  POST /agent/pay                  → pay swap fee with TOTP
```

## Operation Flow

### Step 1: Identify Intent

| User / Agent says | Action |
|---|---|
| "Set up agent payments" | `GET /agent/totp/status` → if not enrolled → `POST /agent/totp/setup` |
| "Authorize agent to pay" | `POST /agent/authorize` |
| "Check agent session" | `GET /agent/authorization-status` |
| Agent needs to pay mid-task | `POST /agent/totp/challenge` → prompt user → `POST /agent/pay` |
| "What has the agent spent?" | `GET /agent/transactions` |
| "Stop the agent" | `POST /agent/revoke` with user's TOTP code |
| "Increase agent limit" | `POST /agent/extend` with fresh TOTP code |

### Step 2: Collect Parameters

| Missing info | Action |
|---|---|
| TOTP not set up | Run setup flow — show QR code, wait for user to scan |
| No spending limit | Ask: "How much total can the agent spend per session?" |
| No per-transaction limit | Ask: "What is the max for a single payment?" |
| No expiry | Ask: "How long should this session last?" |
| No scope | Ask: "What types of payments should the agent be allowed to make?" |
| TOTP code not provided | Show payment details and prompt user to open authenticator app |

### Step 3: Display Results

| Endpoint called | What to show |
|---|---|
| `POST /agent/totp/setup` | QR code image + backup codes (shown once only) |
| `POST /agent/authorize` | Session ID, limit, per-tx cap, scope, expiry |
| `POST /agent/totp/challenge` | Full payment details + TOTP input prompt with countdown |
| `POST /agent/pay` → success | tx hash, amount, remaining session limit |
| `POST /agent/pay` → TOTP_INVALID | "Incorrect code. Please try again." |
| `POST /agent/pay` → TOTP_EXPIRED | "Code expired. Requesting new challenge..." |
| `POST /agent/pay` → CHALLENGE_MISMATCH | "⚠️ Payment details were altered. Transaction blocked." |
| `GET /agent/transactions` | Table: amount, recipient, purpose, TOTP verified, timestamp |
| `POST /agent/revoke` | "Session ended. Total spent: X EDS across N transactions." |

### Step 4: TOTP Gate ⚠️

This gate fires before every `POST /agent/pay`:
```
1. Agent calls POST /agent/totp/challenge  
   → Server issues challengeToken (30s expiry)
   → Server sends push notification to user's device

2. User opens authenticator app
   → Reads 6-digit code for "Luffa Agent Payments"
   → Provides code to agent

3. Agent calls POST /agent/pay with:
   → challengeToken  (from step 1)
   → totpCode        (from step 2)
   → transaction details (must match challenge exactly)

4. Server independently verifies:
   → TOTP code valid for this 30s window? ✅
   → Code not previously used?            ✅
   → tx details match original challenge? ✅
   → Within spending limits?              ✅
   → All pass → transaction executes
   → Any fail → transaction rejected, agent notified
```

**Failure handling**:

| Failure | Agent action |
|---|---|
| `TOTP_INVALID` | Ask user to re-enter code, retry once |
| `TOTP_EXPIRED` | Request new challenge automatically |
| `TOTP_ALREADY_USED` | Request new challenge — warn user of possible replay |
| `CHALLENGE_MISMATCH` | Block immediately — alert user, do not retry |
| 3x consecutive failures | Auto-revoke session, alert user |
